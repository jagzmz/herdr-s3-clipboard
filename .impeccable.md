## Design Context

### Users

Developers using Herdr on macOS, Linux, or Windows. Some are installing their first Herdr plugin; others already understand Herdr actions, panes, and S3-compatible storage. They want to turn a clipboard image into a usable URL with minimal setup and predictable failure behavior.

### Brand Personality

Pragmatic, direct, and trustworthy. Documentation should make security boundaries and configuration precedence explicit without sounding alarmist or forcing experienced readers through introductory material.

### Aesthetic Direction

Developer-first technical documentation with a quick-start path followed by progressively deeper reference material. Prefer copyable commands, compact tables, descriptive headings, and plain language. Avoid marketing clutter, decorative sections, and repeated explanations.

### Design Principles

1. Lead with the shortest successful path from installation to first upload.
2. Keep beginner guidance and expert reference in the same document through progressive disclosure.
3. Explain commands before readers run them, especially when credentials or external storage are involved.
4. Use one consistent term for each concept: plugin config directory, setup wizard, public URL, and presigned URL.
5. Pair every likely failure with a concrete diagnostic or recovery command.
